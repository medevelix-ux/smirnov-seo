Что делает блок: превращает внутренний список задач из Битрикс24 в понятный клиенту список выполненных SEO-работ. Здесь важно не просто показать названия задач, а объяснить пользу: индексация, релевантность, CTR, коммерческие факторы, аналитика.

Откуда брать данные: основной сценарий — найти главную задачу месяца в формате “{Месяц} {Год}. {домен}” и выгрузить все ее подзадачи, включая подзадачи подзадач. По уточнению ТЗ, в отчетный пул попадают все плановые подзадачи главной задачи месяца, даже если часть закрывается в первые дни следующего периода. Зачем нужна предмодерация: в Битриксе могут быть внутренние обсуждения, служебные проверки или задачи, “о которых клиент знать не должен”. Поэтому перед генерацией отчета нужен экран выбора: список задач → крестик “исключить” → кнопка “Продолжить”. Для гостевой ссылки клиент должен видеть только задачи с флагом show_in_guest_report = true.

Минимальные поля: task_id, task_title, task_description, status, comments_text, checklist_items. Дополнительно можно хранить responsible, deadline, time_spent, parent_task_id.

bitrix_source: main_task_title = "{month} {year}. {domain}" project_bitrix_id = project.settings.bitrix_project_id task_selection_flow: 1. load all subtasks from main_task 2. show moderation screen to responsible employee 3. employee excludes internal/client-unwanted tasks 4. save flags: include_in_report, show_in_guest_report 5. send remaining tasks to AI classifier 6. render grouped client-facing block ai_output: task_title_client task_type seo_problem usefulness_comment potential_impact[] effect_type confidence

Что делает ИИ: классифицирует задачи по типам SEO-работ, переводит внутренние формулировки в клиентский язык и объясняет потенциальную пользу. ИИ не имеет права писать, что задача уже дала рост позиций или заявок, если это не подтверждено в других блоках.

Настройка целей: пользователь должен выбрать одну или несколько целей Метрики, сохранить набор для проекта и назначить набор по умолчанию. Цели нужно классифицировать: основные, вспомогательные, технические. В клиентский отчет по умолчанию выводить основные и при необходимости вспомогательные цели.

Важное различие: достижения целей и конверсионные визиты — не одно и то же. Если один визит дал несколько событий, достижения целей могут быть больше числа конверсионных визитов. Для общей конверсии лучше использовать converted_visits / organic_visits, если Метрика позволяет получить такой показатель.

conversion_rate = converted_visits / organic_visits * 100 // fallback: conversion_rate = goal_reaches / organic_visits * 100 goal_types: primary // заявка, телефон, расчет, заказ secondary // каталог, прайс, контакты technical // scroll, popup, open_form warnings: goals_not_selected goal_not_found goal_reaches_zero converted_visits_unavailable goals_set_changed cannot_filter_organic high_traffic_low_conversion: organic_visits >= threshold conversion_rate < avg_seo_conversion_rate * 0.5

Что передавать в ИИ: сводку по выбранным целям, цели с ростом/падением, цели с нулем, страницы с высоким трафиком и низкой конверсией, разбивку по поисковым системам и устройствам. ИИ не должен делать вывод о качестве лидов без CRM.

Что делает блок: формирует два разных слоя: внутренние рекомендации ИИ для SEO-специалиста и обработанный план работ, который можно показывать клиенту по гостевой ссылке. Внутренний слой не должен быть доступен клиенту.

Внутренний слой “Рекомендации к SEO-плану”: видит только SEO-специалист. Здесь отображаются сырые рекомендации ИИ, основание из отчета, затронутые страницы/запросы, источник сигнала, уверенность, чекбоксы выбора и кнопки копирования/переноса.

Клиентский слой “План работ по SEO на следующий месяц”: формируется после обработки рекомендаций специалистом. В гостевой ссылке показывается именно этот слой: понятные формулировки задач без внутренних технических комментариев, без уверенности ИИ и без служебных сигналов.

Кастомный пункт плана: SEO-специалист должен иметь возможность добавить пункт вручную, даже если его не предложил ИИ. Это нужно, когда клиент обозначает новый приоритетный раздел, направление или услугу, и из-за этого план на следующий месяц нужно скорректировать. Такой пункт должен иметь источник manual или client_priority, краткое клиентское описание, приоритет, количество часов и флаг показа в гостевой ссылке.

Откуда брать сигналы: из всех предыдущих блоков. В отличие от общего вывода, сюда нужно передавать детализированные списки: запросы 11–20, запросы 21–30, низкий CTR, падения, страницы с ростом/падением трафика, страницы с высоким трафиком и низкой конверсией, цели с нулем, мобильные проблемы, выполненные задачи текущего месяца.

Как объединять: если несколько сигналов относятся к одной странице, не создавать 3–4 разные задачи. Нужно собрать их в одну комплексную рекомендацию: релевантность + сниппет + коммерческие факторы + конверсия.

Премодерация плана: процесс аналогичен блоку “Выполненные задачи”. Только там специалист исключает лишние задачи из уже выполненных работ, а здесь он выбирает и редактирует будущие задачи. До публикации гостевой ссылки у каждого пункта плана должен быть флаг show_in_guest_report.
