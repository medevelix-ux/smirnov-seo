Инструкция для программиста: выполненные задачи

Что делает блок: превращает внутренний список задач из Битрикс24 в понятный клиенту список выполненных SEO-работ. Здесь важно не просто показать названия задач, а объяснить пользу: индексация, релевантность, CTR, коммерческие факторы, аналитика.

Откуда брать данные: основной сценарий — найти главную задачу месяца в формате “{Месяц} {Год}. {домен}” и выгрузить все ее подзадачи, включая подзадачи подзадач. По уточнению ТЗ, в отчетный пул попадают все плановые подзадачи главной задачи месяца, даже если часть закрывается в первые дни следующего периода. Затем ответственный сотрудник должен вручную исключить задачи, которые не нужно показывать клиенту.

Зачем нужна предмодерация: в Битриксе могут быть внутренние обсуждения, служебные проверки или задачи, “о которых клиент знать не должен”. Поэтому перед генерацией отчета нужен экран выбора: список задач → крестик “исключить” → кнопка “Продолжить”.

Минимальные поля: task_id, task_title, task_description, status, comments_text, checklist_items. Дополнительно можно хранить responsible, deadline, time_spent, parent_task_id.

bitrix_source: main_task_title = "{month} {year}. {domain}" project_bitrix_id = project.settings.bitrix_project_id task_selection_flow: 1. load all subtasks from main_task 2. show moderation screen to responsible employee 3. employee excludes internal/client-unwanted tasks 4. send remaining tasks to AI classifier 5. render grouped client-facing block ai_output: task_title_client task_type seo_problem usefulness_comment potential_impact[] effect_type confidence

Что делает ИИ: классифицирует задачи по типам SEO-работ, переводит внутренние формулировки в клиентский язык и объясняет потенциальную пользу. ИИ не имеет права писать, что задача уже дала рост позиций или заявок, если это не подтверждено в других блоках.

Инструкция для программиста: видимость и позиции

Что делает блок: показывает, как сайт виден в поиске по запросам. Блок помогает понять не только “среднюю позицию”, а структуру спроса: сколько запросов уже в ТОП-10, сколько близко к ТОП-10, где есть падение и где сниппет получает мало кликов.

Основные источники: Яндекс.Вебмастер и Google Search Console. По уточнению ТЗ, главный регион в MVP не используется для Яндекс.Вебмастера и GSC как точный региональный фильтр; он преимущественно нужен для трафика. Если в будущем подключается Webtronica.io как источник целочисленных позиций, тогда регион используется для трекинга позиций.

Приоритетные направления: в настройках проекта нужно хранить список важных страниц/направлений. При агрегации данных по Я.Вебмастеру и GSC система должна выделять запросы и URL, которые относятся к этим направлениям, чтобы клиент видел динамику именно по значимым страницам.

Webtronica.io как опциональный источник: можно использовать для ежедневной динамики целочисленных позиций по фиксированному списку запросов. Эти значения нельзя считать абсолютно точными из-за персонализации выдачи; их лучше использовать как тренд.

position_delta = avg_position_previous - avg_position_current // delta > 0 means improvement // delta < 0 means decline mvp_groups: top_3 = 1..3 top_10 = 1..10 top_30 = 1..30 top_50 = 1..50 quick_growth: avg_position_current >= 11 avg_position_current <= 20 impressions > 0 page_url in priority_directions? boost priority low_ctr: impressions >= threshold avg_position <= 20 ctr < avg_ctr_by_position_group

Что передавать в ИИ: агрегированные сводки по Яндексу/Google, списки запросов 11–20, низкий CTR, падения, рост, предупреждения. Сырые выгрузки целиком не передавать — сначала сгруппировать и отфильтровать.

Инструкция для программиста: поисковый трафик

Что делает блок: показывает динамику именно органического поискового трафика. Это не весь трафик сайта. Из отчета нужно исключить рекламу, прямые заходы, соцсети, мессенджеры, referral, email, UTM-кампании, внутренние переходы, технические визиты и роботов.

Откуда брать данные: Яндекс.Метрика. ID счетчика хранится в настройках проекта. Там же хранится домен, отчетный период, основной регион, настройки исключения роботов и список приоритетных направлений.

Зачем нужен график 12 месяцев: он показывает долгосрочный тренд, сезонность и динамику до/после старта SEO-работ. Дату старта SEO лучше задавать вручную в карточке проекта. На графике выводится отметка “Начало SEO-работ”. Нельзя автоматически писать, что рост произошел именно из-за SEO — допустима осторожная формулировка “после начала работ наблюдается рост”.

metrics: ym:s:visits ym:s:users ym:s:bounceRate ym:s:pageDepth ym:s:avgVisitDurationSeconds dimensions: ym:s:lastSearchEngineRoot ym:s:startURLPath ym:s:deviceCategory ym:s:date ym:s:month filters: traffic_source == organic_search exclude_ads == true exclude_robots == true delta_percent = (current - previous) / previous * 100 search_engine_share = engine_visits / total_organic_visits * 100 avg_before_seo = average(monthly_visits before seo_start_date) avg_after_seo = average(monthly_visits after seo_start_date)

Что должен увидеть программист: блок состоит из нескольких независимых запросов к Метрике: общая сводка, поисковые системы, дни/недели, устройства, посадочные страницы, 12 месяцев. Каждый запрос нужно кэшировать в рамках report_id.

Инструкция для программиста: конверсии

Что делает блок: показывает, как SEO-трафик влияет на целевые действия. Важно анализировать только выбранный набор целей и только органический поиск. Нельзя подмешивать цели из рекламы, прямого трафика или технические события.

Настройка целей: пользователь должен выбрать одну или несколько целей Метрики, сохранить набор для проекта и назначить набор по умолчанию. Цели нужно классифицировать: основные, вспомогательные, технические. В клиентский отчет по умолчанию выводить основные и при необходимости вспомогательные цели.

Важное различие: достижения целей и конверсионные визиты — не одно и то же. Если один визит дал несколько событий, достижения целей могут быть больше числа конверсионных визитов. Для общей конверсии лучше использовать converted_visits / organic_visits, если Метрика позволяет получить такой показатель.

conversion_rate = converted_visits / organic_visits * 100 // fallback: conversion_rate = goal_reaches / organic_visits * 100 goal_types: primary // заявка, телефон, расчет, заказ secondary // каталог, прайс, контакты technical // scroll, popup, open_form warnings: goals_not_selected goal_not_found goal_reaches_zero converted_visits_unavailable goals_set_changed cannot_filter_organic high_traffic_low_conversion: organic_visits >= threshold conversion_rate < avg_seo_conversion_rate * 0.5

Что передавать в ИИ: сводку по выбранным целям, цели с ростом/падением, цели с нулем, страницы с высоким трафиком и низкой конверсией, разбивку по поисковым системам и устройствам. ИИ не должен делать вывод о качестве лидов без CRM.

Инструкция для программиста: общий вывод

Что делает блок: это клиентский итог, а не новый аналитический расчет. Он не должен заново анализировать сырые выгрузки. Его задача — аккуратно объединить уже готовые краткие выводы по задачам, видимости, трафику и конверсиям.

Почему так: каждый предыдущий блок уже интерпретировал свои данные. Если снова передать все сырые таблицы, ИИ может сформулировать противоречивые выводы. Поэтому в общий вывод идут только summary, key_facts, status и warnings.

input_summaries: tasks: status summary key_facts[] warnings[] visibility: status summary key_facts[] warnings[] traffic: status summary key_facts[] warnings[] conversions: status summary key_facts[] warnings[] ai_result: overall_status summary positive_signals[] risks[] data_limitations[] next_month_focus confidence

Правило для ИИ: не придумывать новые цифры, не обещать рост, не утверждать прямую причинно-следственную связь. Допустимо писать “на фоне выполненных работ наблюдается…”.

Инструкция для программиста: рекомендации к SEO-плану

Откуда брать сигналы: из всех предыдущих блоков. В отличие от общего вывода, сюда нужно передавать детализированные списки: запросы 11–20, запросы 21–30, низкий CTR, падения, страницы с ростом/падением трафика, страницы с высоким трафиком и низкой конверсией, цели с нулем, мобильные проблемы, выполненные задачи текущего месяца.
