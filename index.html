Что делает блок: превращает внутренний список задач из Битрикс24 в понятный клиенту список выполненных SEO-работ. Здесь важно не просто показать названия задач, а объяснить пользу: индексация, релевантность, CTR, коммерческие факторы, аналитика.

Откуда брать данные: основной сценарий — найти главную задачу месяца в формате “{Месяц} {Год}. {домен}” и выгрузить все ее подзадачи, включая подзадачи подзадач. По уточнению ТЗ, в отчетный пул попадают все плановые подзадачи главной задачи месяца, даже если часть закрывается в первые дни следующего периода. Затем ответственный сотрудник должен вручную исключить задачи, которые не нужно показывать клиенту.

Зачем нужна предмодерация: в Битриксе могут быть внутренние обсуждения, служебные проверки или задачи, “о которых клиент знать не должен”. Поэтому перед генерацией отчета нужен экран выбора: список задач → крестик “исключить” → кнопка “Продолжить”. Для гостевой ссылки клиент должен видеть только задачи с флагом show_in_guest_report = true.

Минимальные поля: task_id, task_title, task_description, status, comments_text, checklist_items. Дополнительно можно хранить responsible, deadline, time_spent, parent_task_id.

bitrix_source: main_task_title = "{month} {year}. {domain}" project_bitrix_id = project.settings.bitrix_project_id task_selection_flow: 1. load all subtasks from main_task 2. show moderation screen to responsible employee 3. employee excludes internal/client-unwanted tasks 4. save flags: include_in_report, show_in_guest_report 5. send remaining tasks to AI classifier 6. render grouped client-facing block ai_output: task_title_client task_type seo_problem usefulness_comment potential_impact[] effect_type confidence

Что делает ИИ: классифицирует задачи по типам SEO-работ, переводит внутренние формулировки в клиентский язык и объясняет потенциальную пользу. ИИ не имеет права писать, что задача уже дала рост позиций или заявок, если это не подтверждено в других блоках.

09 · Спецификация экспорта
Как программисту генерировать PDF / DOCX / Google Docs
Dev-spec
Не показывать клиенту

            
Принцип: экспортный отчет не повторяет интерактивный дашборд. Все вкладки разворачиваются в последовательные разделы, скрытые блоки гостевой версии не попадают в клиентский документ, а редактируемые ИИ-выводы берутся в версии SEO-специалиста.export_settings:
  export_type: pdf | google_docs | docx
  export_scope: guest_report | internal_report
  include_dev_notes: false
  include_internal_recommendations: false
  include_ai_raw_drafts: false
  expand_tabs_to_sections: true
  page_size: A4
  page_break_mode: section

content_rules:
  tasks.date = closed_at || report_date
  conclusion = seo_conclusion_edited || ai_conclusion_raw
  traffic_recommendations = only show_in_guest_report
  webtronica_positions = summary + link_to_detailed_report
  webtronica_ywm_matches = matched_main_queries + matched_pf_queries + yandex_webmaster_metrics
  backlinks = yandex_webmaster_external_links

            
Правила для форматов

            
PDF
Фиксированный печатный вид A4, номера страниц, разрывы по секциям, без интерактивных элементов.
Google Docs
Редактируемая структура: заголовки H1/H2/H3, таблицы, изображения графиков, клиентские выводы.
DOCX
Файл для отправки клиенту: те же разделы, что PDF, но с возможностью редактирования текста.

            
Edge cases
  • Если источник не подключен — показывать нейтральный блок “данные не подключены”.
  • Если рекомендация скрыта из гостевой ссылки — не экспортировать ее клиенту.
  • Если нет точной связки запрос-конверсия — указывать метод приближения.
  • Если в день несколько сборов позиций — использовать последний сбор дня или настройку проекта.
  • Если ссылочных данных нет — не генерировать оценочные выводы о качестве ссылок.
  • Если общий вывод не отредактирован SEO-специалистом — показать предупреждение перед экспортом.
